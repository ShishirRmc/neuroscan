Take-Home Challenge: AI Vision Classification Prototype
Candidate Brief (2-Day Build Challenge)
Theme: Fine-tune an open-source vision model and ship a trustworthy inference web app
1. Overview
We are evaluating your ability to build an AI-native product end-to-end: data preparation, model fine-tuning, backend APIs, frontend UX, and trustworthy AI exposure. This challenge is intentionally structured to reflect real product engineering work where AI outputs must be understandable, auditable, and safe to use.
You will work with a labeled Kaggle dataset of brain scan images (CT and MRI) and build a prototype application that lets a user upload an image and receive a predicted tumor probability, confidence/uncertainty information, and a clear non-clinical disclaimer.
2. Safety and Product Expectations
* This is a prototype / research demo. It is not a medical device and must not present outputs as a diagnosis.
* We care more about trustworthiness and engineering quality than benchmark accuracy.
* Use AI coding tools freely. We will evaluate your judgment, implementation quality, and your ability to explain tradeoffs.
3. Dataset and Core Task
Dataset: Kaggle - Brain Tumor Multimodal Image (CT and MRI) (brain tumor image classification dataset).
Primary task: Fine-tune an open-source pretrained vision model for image classification (either CT or MRI) and build a web app for image upload and inference.
4. Timebox and Submission Expectations
* Expected effort: Approximately 12 to 16 hours total over 2 days.
* Goal: A thin but complete vertical slice that works end-to-end.
* Do not over-optimize training or infrastructure. Prioritize a working product and clear tradeoffs.
5. Required Deliverables
* README with setup instructions, architecture overview, assumptions, limitations, and tradeoffs
* GitHub repository with source code
* 5 to 10 minute demo video (screen recording)
* Model evaluation summary (metrics and split strategy)
* A runnable local setup (Docker optional but welcome)

6. Challenge Requirements
6.1 Model Training and Evaluation
1. Fine-tune an open-source pretrained vision model (for example: timm backbone, ViT, ResNet, EfficientNet, ConvNeXt, MONAI-based classifier)
2. Implement a reproducible train/validation/test split and describe how you mitigated leakage risk (for example duplicates, near-duplicates, or modality distribution issues)
3. Save a versioned model artifact (for example v0.1.0)
4. Report evaluation metrics appropriate for classification (not just accuracy). Include at least two of: ROC-AUC, precision/recall, F1, sensitivity, specificity, confusion matrix
5. Document thresholding choice or confidence policy if applicable
6.2 Backend API (Inference and Auditability)
6. Build an inference API (FastAPI preferred, equivalent acceptable)
7. Implement image upload inference endpoint (for example POST /predict)
8. Validate file type, file size, and image decode failures with clear error responses
9. Return structured JSON including: predicted_label, tumor_probability, confidence, model_version, inference_time_ms, requires_human_review, timestamp
10. Persist an audit log for inference requests (at minimum: user or mock user, image filename or hash, timestamp, model_version, output summary)
11. Expose a history or audit endpoint/page
6.3 Frontend Web App (Trustworthy UX)
12. Build a web UI (React / Next.js preferred) with image upload and preview
13. Display prediction output with probability and confidence/uncertainty in a clear way
14. Display a prominent disclaimer stating the output is not a clinical diagnosis
15. Avoid overconfident language (for example, do not say 'diagnosed')
16. Flag low-confidence predictions and recommend human review
17. Show inference metadata (model version, timestamp)
18. Provide a simple history/review page of previous inferences
6.4 AI Trust and Product Judgment (Must Have)
19. Implement a confidence policy (for example high/medium/low confidence bands)
20. Set requires_human_review = true for low-confidence outputs (or explain an alternative policy)
21. Clearly communicate limitations (dataset limitations, modality shift, prototype nature)
22. Do not present the system as an autonomous diagnostic tool
